# 06. План реализации, риски и релиз

## 1) Фазы реализации MVP

## Фаза 1: Foundation (недели 1-2)
- Настройка репозитория, окружений, линтинга, базовой структуры client/server.
- Подключение БД и миграций.
- Базовая auth-механика и роли.
- Checkpoint commits: отдельный коммит после каждого завершенного подблока (infra, DB setup, auth baseline).

**Результат:** проект запускается локально, есть auth и каркас API/UI.

## Фаза 2: Core Features (недели 3-5)
- Каталог, карточка курса, уроки/упражнения.
- Запись на курс, прогресс, отправка ответов.
- Базовые страницы student/teacher кабинетов.
- Checkpoint commits: коммит на каждый функциональный блок (catalog, enrollment/progress, lesson/submission, role UI).

**Результат:** ключевой пользовательский путь работает end-to-end.

## Фаза 3: Hardening (недели 6-7)
- Отчеты PDF/DOCX и e-mail (demo/SMTP).
- Аналитика преподавателя, CSV export.
- Усиление валидаций, обработка ошибок, smoke/regression тесты.
- Checkpoint commits: отдельные коммиты для reports, analytics/CSV и hardening-фазы.

**Результат:** продукт готов к пилоту, закрыты критичные риски качества.

## Фаза 4: Release (неделя 8)
- Финальное тестирование, фиксы, release notes.
- Развертывание на production.
- Мониторинг после релиза и стабилизационные исправления.
- Checkpoint commits: релизные фиксы группируются малыми логическими коммитами, без "комбайн"-диффа.

**Результат:** MVP доступен пользователям.

## 2) План-график контрольных точек
- **M1 (конец недели 2):** готов фундамент и auth.
- **M2 (конец недели 5):** готовы core user flows.
- **M3 (конец недели 7):** готовы отчеты, аналитика и hardening.
- **M4 (конец недели 8):** релиз и мониторинг.

## 3) Критические риски и меры снижения
- **R1: Срыв сроков из-за объема backend-функций.**
  - Митигирование: строгий MVP scope, weekly re-prioritization, freeze на новые фичи после M2.
- **R2: Нестабильность отчетов (PDF/DOCX/e-mail).**
  - Митигирование: вынести генерацию в отдельный сервисный слой, ранние интеграционные тесты.
- **R3: Проблемы производительности каталога и аналитики.**
  - Митигирование: индексация, пагинация, оптимизация запросов, профилирование до релиза.
- **R4: Регрессии при частых изменениях.**
  - Митигирование: обязательный CI gate, smoke-checklist перед merge в `main`.
- **R5: Ошибки конфигурации окружений.**
  - Митигирование: env-template, единый runbook, верификация staging перед production.

## 4) Release checklist (Go/No-Go)
- Функции `Must` из требований реализованы.
- Критичные баги (severity high/critical) закрыты.
- Линтер, тесты, сборка проходят.
- Smoke-сценарии ролей guest/student/teacher проходят.
- Отчеты в выбранных форматах создаются корректно.
- Health endpoint и базовый мониторинг работают.
- Документация и known limitations обновлены.
- В ветке нет одного чрезмерно большого несвязного "комбайн-коммита" без обоснования; изменения разбиты на checkpoint commits.

## 5) Пострелизный контроль (первые 7 дней)
- Мониторинг ошибок API и времени ответа.
- Контроль пользовательских сценариев первого входа/обучения.
- Приоритизация bugfix backlog на основе реальных инцидентов.

## 6) Критерий готовности этапа "Delivery Roadmap"
- Согласованы фазы, контрольные точки и критерии перехода.
- У команды есть общий release checklist и правила Go/No-Go.
