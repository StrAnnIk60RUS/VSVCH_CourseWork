# 07. Task template (anti-random)

Этот шаблон обязателен для постановки задач Cursor в проекте VSVH.
Цель - исключить размытые границы, случайные расширения и несогласованные изменения.

## Как использовать

- Заполнять все разделы перед началом реализации.
- Если поле не применимо, указывать `N/A` и краткую причину.
- Если по задаче есть архитектурная развилка, решение фиксируется в `08-decisions-log.md`.

---

## Task Card

### Goal
- Что именно должно быть достигнуто в конце задачи.

### InScope
- Какие изменения разрешены.
- Какие модули/файлы можно затрагивать.

### OutOfScope
- Что явно запрещено менять в рамках этой задачи.

### NonGoals
- Что может быть полезно, но намеренно НЕ делается сейчас.

### AcceptanceCriteria
- Проверяемые пункты готовности (Given/When/Then или чеклист).

### AffectedRequirements
- Указать ID требований из `03-architecture-and-api.md` §17 (`R*`).
- Указать связь с контрактами из `docs/FUNCTIONAL_REQUIREMENTS.md` и страницами из `docs/PAGES_AND_FEATURES.md` (если применимо).

### Priority
- Указать приоритет из `03-architecture-and-api.md` §19: `P0`, `P1`, `P2` или `P3`.
- Кратко обосновать, почему задача выполняется сейчас.

### ChangeBudget
- Лимит на изменение:
  - `maxFilesChanged`:
  - `maxModulesChanged`:
  - `forbiddenRefactors`: (например, массовые переименования, переносы директорий, форматирование всего проекта)
- Выход за budget требует отдельного подтверждения.

### CommitPlan
- Ожидаемое число коммитов по задаче:
- Границы каждого коммита:
  - `commit1_scope` (что входит / что не входит)
  - `commit2_scope` (при необходимости)
  - `commitN_scope`
- Типы Conventional Commit для каждого инкремента:
  - `feat` | `fix` | `docs` | `refactor` | `test` | `chore`
- Правило checkpoint:
  - После завершения каждого логического блока и локальной проверки исполнитель предлагает коммит пользователю.

### TestPlan
- Что будет проверено локально:
  - lint/typecheck/tests/build
  - ручной smoke для затронутого сценария
- Что обязательно проверить в UI/API после изменения.

### RisksAndRollback
- Основные риски изменения.
- План отката (rollback) при регрессии.

### Deliverables
- Какие артефакты должны появиться:
  - код
  - тесты
  - обновленная документация
