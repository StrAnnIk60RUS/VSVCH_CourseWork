# 05. Инженерные процессы и качество

## 1) Git-процесс
- Основная ветка: `main`.
- Рабочие ветки: `feature/<name>`, `fix/<name>`, `chore/<name>`.
- Каждый значимый change через Pull Request.
- PR должен содержать: цель, список изменений, риски, шаги проверки.

## 2) Коммиты и ревью
- Формат коммитов: Conventional Commits (`feat`, `fix`, `docs`, `refactor`, `test`, `chore`).
- Маленькие атомарные коммиты, отражающие единый смысл изменения.
- Минимум 1 reviewer на PR в MVP-команде.
- Merge только при зеленых checks (lint/test/build).
- Коммит-сообщения не содержат ссылок на инструменты/ассистентов (включая Cursor).

### Checklist разумного коммита
- Коммит закрывает одну логическую цель.
- Diff обозримый для ревью и не содержит несвязанных изменений.
- Для затронутого контура пройдены обязательные проверки (lint/test/typecheck/build по применимости).
- Сообщение коммита объясняет смысл изменения, а не только список файлов.
- Коммит не смешивает несвязанные `refactor + feature` без явного обоснования.
- В тексте коммита нет упоминаний инструмента разработки (например, Cursor).

## 3) Стандарты кода
- TypeScript strict mode для client/server (где применимо).
- ESLint + Prettier как обязательный минимум.
- Единый стиль импорта и нейминга.
- Запрет хардкода секретов в коде.
- Бизнес-правила только в service-слое, UI-компоненты без тяжелой доменной логики.

## 4) Тестовая стратегия MVP

### Backend
- Unit-тесты критичных сервисов:
  - auth
  - enrollments/progress recalculation
  - submissions scoring
- Integration-тесты ключевых endpoint:
  - auth flow
  - course/lesson/exercise permissions
  - enroll/unenroll
  - reports generation endpoints

### Frontend
- Unit-тесты для утилит и бизнес-хуков.
- Компонентные тесты для ключевых сценариев:
  - каталог и фильтры
  - форма входа/регистрации
  - прогресс/статусы загрузки

### Smoke (pre-release)
- Базовый сценарий гостя, студента и преподавателя.
- Проверка health endpoint.
- Проверка генерации минимум одного отчета.

## 5) CI/CD minimum checks
- `lint` для client/server.
- `test` (минимальный набор обязательных тестов).
- `build` (frontend и backend, если применимо).

## 6) Definition of Ready / Definition of Done

### DoR
- Есть ясная постановка и критерии приемки.
- Понятны зависимости и затрагиваемые модули.
- Согласован контракт API/данных (если релевантно).

### DoD
- Реализован сценарий и edge-cases из задачи.
- Обновлены/добавлены тесты.
- Документация актуализирована.
- PR прошел ревью и все checks.

## 7) Критерий готовности этапа "Engineering Process"
- Команда работает по единому git-процессу.
- Кодовые стандарты и тестовые требования зафиксированы.
- Есть минимальный quality gate перед merge/release.

## 8) Обязательные anti-random артефакты

Для детерминированного исполнения задач Cursor в рамках engineering процесса обязательны:

- `docs/preproject/07-task-template.md` - шаблон постановки задачи с `InScope/OutOfScope/ChangeBudget`.
- `docs/preproject/08-decisions-log.md` - журнал ADR-lite решений.
- `docs/preproject/09-known-pitfalls.md` - каталог типовых проблем и превентивных правил.
- `docs/preproject/10-quality-gates.md` - единый pipeline и gate-условия.

Без использования этих артефактов задача не считается process-compliant.
