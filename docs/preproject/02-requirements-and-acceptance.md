# 02. Требования и критерии приемки

## 1) Функциональные области MVP

Этот документ фиксирует обязательный функционал до старта активной разработки. Подробные API-требования уже описаны в `docs/FUNCTIONAL_REQUIREMENTS.md`, а карта страниц в `docs/PAGES_AND_FEATURES.md`.

### Блок A. Доступ и роли
- Регистрация, вход, получение текущего пользователя.
- Роли: `student`, `teacher`.
- Защищенные действия только по JWT.

**Критерии приемки:**
- При корректной регистрации возвращается token и профиль.
- При неверных данных входа возвращается `401`.
- При отсутствии токена защищенные эндпоинты возвращают `401`.

### Блок B. Каталог и курсы
- Публичный каталог курсов с фильтрацией, сортировкой, пагинацией.
- Карточка курса с преподавателем, уроками, количеством упражнений.
- Для преподавателя: создание/обновление/удаление только своих курсов.

**Критерии приемки:**
- Фильтры и сортировки дают предсказуемый результат.
- Курс чужого преподавателя нельзя изменить (`403`).
- Пустая выборка корректно возвращает `items: []`.

### Блок C. Обучение студента
- Запись на курс/отписка.
- Прохождение уроков, отметка завершения.
- Отправка ответов с автопроверкой и начислением баллов.
- Пересчет прогресса после действий студента.

**Критерии приемки:**
- Повторная запись на курс невозможна (`409`).
- Отметка урока пересчитывает прогресс.
- Ответы на упражнения дают балл согласно правилам нормализации.

### Блок D. Персональные функции
- Избранное: список/добавление/удаление.
- Напоминания: CRUD для владельца.
- Профиль: чтение/обновление имени.

**Критерии приемки:**
- Пользователь видит только свои данные.
- Операции над чужими сущностями запрещены.

### Блок E. Преподавательская аналитика и отчеты
- Список своих курсов и студентов.
- Фильтр активности, сортировка студентов.
- Экспорт CSV.
- Отчеты PDF/DOCX: студентский и по курсу.
- Отправка отчетов на e-mail (SMTP или demo mode).

**Критерии приемки:**
- CSV выгружается с корректными колонками.
- Отчеты скачиваются в выбранном формате.
- При отсутствии SMTP API не падает, а возвращает demo-результат.

## 2) Нефункциональные требования (NFR)
- **Безопасность:** bcrypt для паролей, JWT, CORS, валидация входных данных, rate limiting.
- **Производительность:** каталог с пагинацией, ограничение `limit`, запросы без N+1 для ключевых списков.
- **Надежность:** health endpoint, обработка ошибок, недопущение падений на внешних интеграциях.
- **Масштабируемость:** разделение слоев controller/service/repository, миграции БД.
- **Доступность:** базовые требования A11y (фокус, label, клавиатурная навигация, контраст).
- **Наблюдаемость:** структурированные логи ошибок и базовые метрики API.

## 3) Матрица "что считаем готовым" (DoR/DoD Lite)

### Definition of Ready (до старта задачи)
- Есть user story и бизнес-цель.
- Есть критерии приемки.
- Понятен контракт API или UI-поведение.
- Нет блокирующих зависимостей.

### Definition of Done (после выполнения)
- Код соответствует требованиям и проходит линтер/тесты.
- Есть проверка edge-cases из критериев приемки.
- Обновлена документация (если изменен контракт/поведение).
- Изменение покрыто ревью и готово к интеграции.

## 4) Приоритеты реализации
- **Must:** auth, каталог, запись, уроки/упражнения, базовые отчеты.
- **Should:** расширенная аналитика преподавателя, e-mail отправка отчетов.
- **Could:** роль admin и расширения UX.

## 5) Критерий готовности этапа "Requirements"
- Зафиксирован согласованный список функциональных и нефункциональных требований.
- Для критичных сценариев определены проверяемые критерии приемки.
