# 04. Execution workflow (обязательный порядок работы)

Этот документ фиксирует, как выполнять задачи в VSVH, чтобы реализация всегда соответствовала мастер-плану:
`docs/preproject/03-architecture-and-api.md`.

## 1) Статус мастер-плана

- Мастер-план проекта: `docs/preproject/03-architecture-and-api.md`.
- Любая задача должна быть привязана к gap list и приоритетам из раздела `§19`.
- Оценка готовности ведется по фактическому состоянию (`§1.2`, `§17`), а не по целевому описанию.

## 2) Intake перед стартом задачи (обязательно)

Перед началом реализации пройти три проверки:

1. `§1.2` — определить текущий фактический статус затронутого требования.
2. `§17` — определить конкретный requirement ID (R1...R13), который закрывается задачей.
3. `§19` — зафиксировать приоритет (P0/P1/P2/P3) и объяснить, почему задача выполняется сейчас.

Минимальный формат фиксации в описании задачи/PR:

- `Affected requirement`: `R*`
- `Priority`: `P*`
- `Target delta`: какой статус ожидается после изменения (`Missing -> Partial`, `Partial -> Implemented`).

### Pre-Start Compliance checklist

Перед началом реализации дополнительно подтвердить:

- мастер-план и приоритеты сверены (`§1.2`, `§17`, `§19`);
- шаблон задачи из `07-task-template.md` заполнен;
- `CommitPlan` определен для текущей задачи;
- commit message policy учтена (без упоминаний инструмента/ассистента, включая Cursor);
- quality gates из `10-quality-gates.md` определены для текущего инкремента.

## 3) Реализация (во время задачи)

- Не нарушать архитектурные слои, описанные в `03-architecture-and-api.md` (`routes -> controllers -> services -> data access`).
- Для ролевых сценариев проверять минимум `student` и `teacher`.
- Для пользовательских параметров UI (фильтры/сортировки/пагинация/избранное) использовать localStorage policy из `§12`.

## 4) DoD-gate перед закрытием задачи

Перед переводом задачи в done:

1. Проверить Definition of Done `§16` из `03-architecture-and-api.md`.
2. Убедиться, что изменение не ухудшило матрицу `§17`.
3. Для финального релизного цикла отдельно применить gate из `§20`.

Без прохождения DoD-gate задача считается незавершенной.

## 5) Синхронизация документации (обязательно для API/страниц)

Если затронуты API-контракты или страницы:

- Обновить `docs/FUNCTIONAL_REQUIREMENTS.md` (контракты, критерии приемки, FR-ID).
- Обновить `docs/PAGES_AND_FEATURES.md` (маршруты, роли, UI-сценарии, связь с API).
- Обновления делать в рамках той же задачи, без отложенной "документации потом".

## 6) Готовность к сдаче

Итоговая готовность курсовой подтверждается только при выполнении `§20` из мастер-плана и отсутствии критичных пробелов по `§17`.
