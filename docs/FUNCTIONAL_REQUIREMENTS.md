## 1. Термины и сокращения

| Термин | Определение |
|--------|-------------|
| Пользователь | Учётная запись с уникальным e-mail, именем и хешем пароля. |
| Роль | Именованный набор прав: `student`, `teacher`. Назначается связью many-to-many с пользователем. |
| JWT | Токен доступа, выдаваемый при регистрации/входе, срок жизни 7 суток (если иное не задано конфигурацией). |
| Курс | Сущность с штатом в `course_staff` (преподаватель, автор, методист, куратор), названием, описанием, языком, уровнем; рейтинг в UI опирается на отзывы (`course_reviews`), в таблице курса кэшируется `rating_average` для сортировки; число записей — по `enrollments`. |
| Урок | Элемент курса с порядком, заголовком, текстовым содержимым, опциональной ссылкой на видео. |
| Упражнение | Задание внутри урока: формулировка вопроса, эталонный ответ, максимальный балл. |
| Запись (Enrollment) | Связь студент–курс с прогрессом в процентах и датой записи; одна запись на пару пользователь–курс. |
| Отправка ответа (Submission) | Попытка студента с текстом ответа и начисленным баллом. |
| Завершение урока | Факт прохождения урока пользователем; влияет на прогресс записи. |
| Избранное | Пара пользователь–курс для быстрого доступа к курсу. |
| Напоминание | Запись с заголовком и датой/временем срабатывания, принадлежащая пользователю. |

## 2. Заинтересованные стороны и роли

### 2.1 Студент (`student`)

- Просматривает каталог и детали курсов, записывается на курсы, снимает запись.
- Проходит уроки, отмечает завершение урока (при активной записи), решает упражнения с автопроверкой.
- Управляет избранным и напоминаниями (в рамках своей учётной записи).
- Просматривает историю своих отправок; формирует отчёт о собственном прогрессе (PDF/DOCX) и может запросить отправку на e-mail (при настроенном SMTP — реальная отправка; иначе — демо-поведение).

### 2.2 Преподаватель (`teacher`)

- Создаёт, изменяет и удаляет **свои** курсы.
- Управляет уроками и упражнениями внутри **своих** курсов.
- Просматривает список своих курсов со счётчиками уроков и записей.
- Просматривает список студентов по курсу с прогрессом, активностью, фильтрацией и сортировкой; экспортирует список в CSV.
- Формирует сводку по группе курса (PDF/DOCX) и может запросить отправку на e-mail.

Требования ниже относятся к ролям `student` и `teacher`. Роль администратора в маркетинговом описании может упоминаться, но **в текущей реализации не заведена**; планируется **FR-ADM-01:** поддержка роли `admin` и отдельная спецификация прав.

### 2.3 Неаутентифицированный пользователь

- Имеет доступ к публичному каталогу курсов и к чтению списка уроков и упражнений курса (как в текущем API).
- Не выполняет операции, требующие JWT, за исключением регистрации и входа.

## 3. Общие бизнес-правила

| ID | Правило |
|----|---------|
| BR-01 | Идентификация пользователя в защищённых операциях выполняется по JWT в заголовке `Authorization: Bearer <token>`. |
| BR-02 | При отсутствии токена или при недействительном токене API возвращает ответ `401` с понятным сообщением об ошибке. |
| BR-03 | При недостаточных правах для роли API возвращает ответ `403`. |
| BR-04 | Прогресс записи на курс вычисляется как округлённый процент: `(число завершённых уроков / число уроков курса) × 100`; при нуле уроков прогресс = 0%. |
| BR-05 | После завершения урока и после отправки ответа на упражнение выполняется пересчёт прогресса для соответствующей записи студента на курс. |
| BR-06 | Автопроверка ответа: сравнение выполняется после нормализации (обрезка пробелов, приведение к нижнему регистру, схлопывание внутренних пробелов в один). Регистр и лишние пробелы не должны влиять на результат. |
| BR-07 | При полной правильности ответа начисляется `maxScore` упражнения, иначе 0. |
| BR-08 | Один студент не может иметь две записи на один и тот же курс (уникальность пары userId + courseId). |
| BR-09 | Преподаватель может изменять только курсы, где он присутствует в `course_staff` с ролью, допускающей редактирование контента (как минимум `TEACHER`; при необходимости — и другие роли по политике продукта). |
| BR-10 | Число записавшихся на курс определяется агрегатом `COUNT(*)` по таблице `enrollments` для данного курса (денормализованный счётчик в таблице курсов не хранится, 3НФ). |
| BR-11 | Пагинация каталога курсов: параметр `limit` ограничивается сверху значением 50 даже при большем запросе. |

## 4. Функциональные требования по областям

Каждое требование имеет: **ID**, формулировку в повелительном наклонении («система должна…», «пользователь должен иметь возможность…»), **критерии приёмки** (проверяемые условия), при необходимости — примечание о текущей реализации.

### 4.1 Аутентификация и регистрация

#### FR-AUTH-01 — Регистрация пользователя

**Описание:** Система должна позволять создать учётную запись с e-mail, паролем, именем и опциональным выбором роли при регистрации.

**Детали:**

- Обязательные поля тела запроса: `email`, `password`, `name`. При отсутствии любого из них — `400` с текстом, указывающим на необходимость указать email, пароль и имя.
- При совпадении e-mail с существующим пользователем — `409` «Пользователь уже существует».
- Роль: если передано `role === 'teacher'`, назначается роль преподавателя; иначе — студент.
- Пароль должен храниться только в виде криптостойкого хеша (bcrypt).
- Успех: `201`, тело содержит `token` (JWT) и объект `user` с `id`, `email`, `name`, `roles`.

**Критерии приёмки:**

1. Успешная регистрация создаёт пользователя в БД и возвращает валидный JWT.
2. Повторная регистрация с тем же e-mail отклоняется с `409`.
3. Невалидный/неполный ввод отклоняется с `400`.

#### FR-AUTH-02 — Вход в систему

**Описание:** Система должна аутентифицировать пользователя по e-mail и паролю.

**Детали:**

- Обязательны `email` и `password`; иначе `400`.
- Неверная пара учётных данных — `401` без раскрытия, существует ли пользователь (единое сообщение «Неверные учётные данные»).
- Успех: `200`, `token` и `user` как при регистрации.

**Критерии приёмки:**

1. Корректные учётные данные возвращают JWT и профиль.
2. Некорректные — `401`.

#### FR-AUTH-03 — Получение текущего пользователя по токену

**Описание:** Система должна позволять получить профиль по Bearer-токену (эндпоинт «me» в модуле auth).

**Детали:**

- Нет заголовка или невалидный токен — `401`.
- Пользователь удалён из БД — `401` «Пользователь не найден».

**Критерии приёмки:**

1. Валидный токен возвращает актуальные `id`, `email`, `name`, `roles`.

### 4.2 Профиль пользователя (`/api/users`)

#### FR-USER-01 — Чтение собственного профиля

**Описание:** Аутентифицированный пользователь должен иметь возможность получать свои данные: `id`, `email`, `name`, `createdAt`.

**Критерии приёмки:** `GET` с валидным JWT возвращает `200` и указанные поля; без токена — `401`.

#### FR-USER-02 — Обновление отображаемого имени

**Описание:** Пользователь должен иметь возможность изменить своё имя (`name`).

**Детали:** Только поле `name` из тела запроса; остальные атриуты профиля через этот эндпоинт не меняются.

**Критерии приёмки:** После `PUT` возвращается обновлённый `name`; неаутентифицированный запрос — `401`.

---

### 4.3 Каталог и карточка курса (`/api/courses`)

#### FR-COURSE-01 — Просмотр каталога с фильтрацией, сортировкой и пагинацией

**Описание:** Система должна отдавать список курсов с поддержкой:

- Фильтры query: `language`, `level`, `minRating` (минимальный рейтинг, число), `search` (поиск без учёта регистра по названию и описанию).
- Сортировка: `sort` ∈ {`createdAt`, `popularity`, `rating`} (где `popularity` = число строк в `enrollments` для курса), `order` ∈ {`asc`, `desc`}; по умолчанию сортировка по дате создания убыв.
- Пагинация: `page` (по умолчанию 1), `limit` (по умолчанию 8, максимум 50).

**Ответ:** `items`, `total`, `page`, `limit`; в элементах — данные курса, преподаватель (`id`, `name`), число уроков.

**Критерии приёмки:**

1. Комбинации фильтров сужают выборку корректно.
2. `limit=999` фактически не превышает 50.
3. Пустая выборка возвращает `items: []` и `total: 0`.

#### FR-COURSE-02 — Просмотр деталей курса

**Описание:** По идентификатору курса система должна возвращать полную карточку: преподаватель, уроки в порядке `order`, для каждого урока — число упражнений.

**Критерии приёмки:** Несуществующий id — `404` «Курс не найден».

#### FR-COURSE-03 — Создание курса (преподаватель)

**Описание:** Преподаватель должен иметь возможность создавать курс; создатель автоматически добавляется в `course_staff` (например, роль `TEACHER`); назначать чужих пользователей в штат курса через API без прав администратора нельзя.

**Детали полей:** `title`, `description` (по умолчанию пустая строка), `language` (по умолчанию `English`), `level` (по умолчанию `A1`), `rating` опционально (число).

**Критерии приёмки:** Без роли teacher — `403`; без авторизации — `401`; успех — `201`.

#### FR-COURSE-04 — Изменение курса (владелец)

**Описание:** Преподаватель должен иметь возможность обновлять только свой курс.

**Критерии приёмки:** Чужой курс или отсутствие пользователя в `course_staff` с правом правки — `403`; частичное обновление допустимо (переданные поля обновляются).

#### FR-COURSE-05 — Удаление курса (владелец)

**Описание:** Преподаватель должен иметь возможность удалять только свой курс.

**Критерии приёмки:** Успех — `204`; чужой курс — `403`.

### 4.4 Уроки (`/api/courses/:courseId/lessons`)

#### FR-LESSON-01 — Публичное чтение списка уроков курса

**Описание:** Любой клиент должен иметь возможность получать список уроков курса с упражнениями, отсортированный по `order` возрастанию.

**Критерии приёмки:** Порядок уроков соответствует полю `order`.

#### FR-LESSON-02 — Создание урока (преподаватель-владелец курса)

**Описание:** Преподаватель должен иметь возможность создавать урок в своём курсе.

**Детали:** `title` по умолчанию «Урок», `content` по умолчанию пусто, `videoUrl` опционально; `order` если не задан — max(order)+1 по курсу.

**Критерии приёмки:** Не владелец — `403`; нет курса — `404`.

#### FR-LESSON-03 — Обновление и удаление урока

**Описание:** Только владелец курса должен иметь возможность изменять и удалять уроки; идентификатор урока должен принадлежать указанному `courseId`.

**Критерии приёмки:** Успешное удаление — `204`; нарушение прав — `403` или `404` согласно реализации проверки.

### 4.5 Упражнения (`/api/courses/:courseId/lessons/:lessonId/exercises`)

#### FR-EX-01 — Публичное чтение списка упражнений урока

**Описание:** Клиент должен иметь возможность получать список упражнений по `lessonId` (в контексте курса и урока в URL).

#### FR-EX-02 — CRUD упражнений преподавателем

**Описание:** Преподаватель должен иметь возможность создавать, обновлять и удалять упражнения только в уроках своих курсов.

**Детали создания:** `question` по умолчанию пусто, `correctAnswer` приводится к строке, `maxScore` по умолчанию 10.

**Критерии приёмки:** Проверка владения через цепочку урок → курс → `course_staff` (роль с правом редактирования).

### 4.6 Запись на курс и прогресс (`/api/enrollments`)

#### FR-ENR-01 — Запись студента на курс

**Описание:** Студент должен иметь возможность создать запись на существующий курс.

**Детали:** Тело: `courseId`. Курс не найден — `404`. Уже записан — `409`. При успехе создаётся запись с `progress: 0` (число записавшихся выводится запросом к `enrollments`).

**Критерии приёмки:** Не студент — `403`; создание записи в одной транзакции с проверкой уникальности пары пользователь–курс.

#### FR-ENR-02 — Список своих записей

**Описание:** Аутентифицированный пользователь должен иметь возможность получать список своих записей с данными курса и именем преподавателя.

#### FR-ENR-03 — Отписка от курса

**Описание:** Студент должен иметь возможность удалить свою запись по `courseId`; при отсутствии записи — `404`; при успехе агрегат по `enrollments` для курса уменьшается на единицу автоматически.

#### FR-ENR-04 — Отметка урока как пройденного

**Описание:** Студент должен иметь возможность зафиксировать завершение урока, если записан на курс.

**Детали:** `POST .../complete-lesson/:courseId/:lessonId`. Нет записи на курс — `403` «Сначала запишитесь на курс». Повторная отметка идемпотентна (upsert без изменения даты при повторе — поведение БД `update: {}`).

**Критерии приёмки:** Ответ содержит пересчитанный `progress` в процентах.

### 4.7 Отправка ответов на упражнения (`/api/submissions`)

#### FR-SUB-01 — Отправка ответа студентом

**Описание:** Студент должен иметь возможность отправить ответ на упражнение, будучи записанным на курс, к которому относится упражнение.

**Детали:** Тело: `exerciseId`, `answer`. Нет записи на курс — `403`. Упражнение не найдено — `404`. Сохраняется запись с баллом и флагом корректности в ответе (`correct`).

**Критерии приёмки:** Нормализация строк согласно BR-06; после сохранения вызывается пересчёт прогресса записи.

#### FR-SUB-02 — История отправок пользователя

**Описание:** Аутентифицированный пользователь должен иметь возможность получать до 200 последних отправок с данными упражнения и урока, сортировка по дате убыванию.

### 4.8 Избранное (`/api/favorites`)

#### FR-FAV-01 — Список избранных курсов

**Описание:** Пользователь должен иметь возможность получать список курсов из своего избранного (как массив объектов курса с преподавателем).

#### FR-FAV-02 — Добавление в избранное

**Описание:** Пользователь должен иметь возможность добавить курс в избранное; повторное добавление идемпотентно (upsert).

**Критерии приёмки:** `201`, `{ ok: true }`.

#### FR-FAV-03 — Удаление из избранного

**Описание:** Пользователь должен иметь возможность удалить курс из избранного; успех — `204`.

---

### 4.9 Напоминания (`/api/reminders`)

#### FR-REM-01 — Список напоминаний

**Описание:** Пользователь должен видеть только свои напоминания, сортировка по `remindAt` возрастанию.

#### FR-REM-02 — Создание напоминания

**Описание:** Обязательны `title` и `remindAt` (дата/время, парсится как `Date`); иначе `400`.

#### FR-REM-03 — Изменение и удаление

**Описание:** Только владелец; чужой или несуществующий id — `404`; успешное удаление — `204`.

---

### 4.10 Кабинет преподавателя (`/api/teacher`)

#### FR-TCH-01 — Мои курсы

**Описание:** Преподаватель должен иметь возможность получать список курсов, где он владелец, с числом записей и числом уроков, сортировка по дате создания убыванию.

#### FR-TCH-02 — Студенты курса (аналитика)

**Описание:** Преподаватель должен иметь возможность получать табличные данные по студентам курса: имя, e-mail, прогресс, дата записи, последняя активность (max дата отправки по пользователю или дата записи), признак `active` (активен, если с последней активности прошло не более 14 суток).

**Фильтр query:** `status` ∈ {`active`, `inactive`} — фильтрация по признаку активности.

**Сортировка query:** `sort` ∈ {`name`, `progress`, `activity`}, `order` ∈ {`asc`, `desc`}; сортировка по имени с учётом локали `ru`.

**Критерии приёмки:** Чужой курс — `403`.

#### FR-TCH-03 — Экспорт студентов в CSV

**Описание:** Преподаватель должен иметь возможность скачать CSV с колонками: name, email, progress, enrolledAt для своего курса.

**Детали:** Кодировка UTF-8 с BOM (`\uFEFF`) для корректного открытия в Excel; экранирование кавычек в имени по стандарту CSV.

**Критерии приёмки:** `Content-Type: text/csv; charset=utf-8`, заголовок `Content-Disposition: attachment`.

### 4.11 Отчёты (`/api/reports`)

#### FR-RPT-01 — PDF прогресса студента

**Описание:** Студент должен иметь возможность скачать PDF с ФИО, e-mail, периодом (текущий год в данных), таблицей курсов с прогрессом % и суммой баллов по курсу (по отправкам).

**Критерии приёмки:** Роль student; нет пользователя — `404`.

#### FR-RPT-02 — DOCX прогресса студента

**Описание:** Аналогично FR-RPT-01 в формате DOCX.

#### FR-RPT-03 — PDF сводки по курсу (преподаватель)

**Описание:** Преподаватель должен иметь возможность скачать PDF по query `courseId`: название курса, язык, уровень, число студентов, средний прогресс (целое округление), список студентов с прогрессом и датой последней активности (формат YYYY-MM-DD).

**Критерии приёмки:** Без `courseId` — `400`; чужой курс — `403`.

#### FR-RPT-04 — DOCX сводки по курсу

**Описание:** Аналогично FR-RPT-03 в формате DOCX.

#### FR-RPT-05 — Отправка отчёта на e-mail

**Описание:** Аутентифицированный пользователь должен иметь возможность инициировать отправку отчёта на указанный адрес.

**Тело:** `email`, `type`, `format`, опционально `courseId` для сводки преподавателя.

- `type === 'student-progress'`: только при роли student; `format` ∈ {`pdf`, `docx`}.
- `type === 'course-summary'`: только при роли teacher и наличии `courseId`; `format` ∈ {`pdf`, `docx`}.

**Поведение при отсутствии SMTP:** отсутствие настройки почты не должно приводить к аварийному завершению запроса; возвращается объект с `demo: true` и пояснением, что письмо «залогировано» (демо-режим). При настроенном SMTP — реальная отправка вложения, ответ `sent: true`.

**Критерии приёмки:** Некорректные комбинации type/role/courseId дают `400` или `403`; неверный `type` — `400`.

## 5. Данные и целостность (сводка по сущностям)

Требования к данным (длины строк, допустимые значения `level`/`language` как справочники) в текущей реализации **не жёстко валидируются на уровне API** (строки свободного формата). Для промышленной эксплуатации рекомендуется расширение:

- **FR-DATA-01 (рекомендуемое):** валидация уровня CEFR (A1–C2) и кодов языков по whitelist на уровне API или БД.

## 6. Нефункциональные ожидания (кратко, для трассировки)

Эти пункты не дублируют полную НФТ-спецификацию, но фиксируют ожидания, связанные с функциональностью:

- **Безопасность:** пароли не возвращаются в API; JWT подписан секретом из конфигурации; CORS ограничен `CLIENT_ORIGIN`.
- **Наблюдаемость:** ошибки сервера логируются; демо-отправка писем логируется в консоль при отсутствии SMTP.
- **Производительность:** история отправок ограничена 200 записями; каталог — пагинация и cap на `limit`.

## 7. Состояние реализации клиентской части

В репозитории реализованы сервер (Express + Sequelize + PostgreSQL) и набор UI-компонентов. **Полноценное покрытие всех FR веб-интерфейсом в одном главном приложении не подтверждено структурой `client/src` на момент составления документа.** Требования разделов 4–5 считаются **обязательными для backend API**; требования к UI формулируются как:

- **FR-UI-01 (продукт):** пользовательский интерфейс должен предоставлять доступ ко всем операциям, соответствующим роли, описанным в разделе 4, без обхода бизнес-правил.
- **FR-UI-02 (продукт):** интерфейс должен быть адаптивным (desktop / tablet / mobile) и соответствовать заявленной дизайн-системе (токены, контраст, фокус, клавиатура).

Детализация экранов и user stories UI может выноситься в отдельный документ UX/UI по мере готовности макетов.
