Перед любыми изменениями в проекте VSVH:

1. Используй `docs/preproject/03-architecture-and-api.md` как мастер-план проекта и единый источник приоритетов реализации.
2. Учитывай и соблюдай требования из `docs/preproject/03-architecture-and-api.md` (целевое ТЗ, фактический аудит §1.2, gap list §19, DoD §20).
3. Выполняй задачи по процессу из `docs/preproject/04-execution-workflow.md` (intake, DoD-gate, синхронизация документации).
4. Перед стартом любой задачи явно сверяй разделы §1.2, §17 и §19 и фиксируй, какой gap/priority закрывает изменение.
5. Перед завершением задачи проверяй Definition of Done: §16 для task-level и §20 перед финальной сдачей/релизом.
6. Если изменение затрагивает API или страницы, синхронно обновляй `docs/FUNCTIONAL_REQUIREMENTS.md` и `docs/PAGES_AND_FEATURES.md` в рамках той же задачи.
6a. Если изменение затрагивает схему БД, синхронно обновляй `server/database/migrations/init-schema.sql`, миграцию Sequelize, модели в `server/db/models/`, при необходимости seed и `docs/DATABASE.md` (и §5 в `FUNCTIONAL_REQUIREMENTS.md`).
7. Не смешивай «как должно быть по документации» и «что уже есть в коде» — ориентируйся на §1.2 при оценке готовности.
8. При затронутых API/страницах сверяйся с `docs/FUNCTIONAL_REQUIREMENTS.md` и `docs/PAGES_AND_FEATURES.md`.
9. Перед началом задачи заполняй обязательный шаблон из `docs/preproject/07-task-template.md` с `InScope`, `OutOfScope`, `AcceptanceCriteria`, `ChangeBudget`.
10. Не выполняй незапрошенные рефакторы (массовые переименования, переносы директорий, форматирование вне scope) без явного запроса.
11. Не изменяй `OutOfScope`-области задачи и не выходи за `ChangeBudget` без подтверждения.
12. При наличии 2+ равновалидных архитектурных вариантов останавливайся и запрашивай подтверждение выбранного пути до реализации.
13. Архитектурные и процессные развилки фиксируй в `docs/preproject/08-decisions-log.md`.
14. Перед завершением задачи проходи quality gates из `docs/preproject/10-quality-gates.md`.
15. Проверяй типовые риски по `docs/preproject/09-known-pitfalls.md` перед передачей результата.
16. В финальном отчете по задаче всегда указывай: что изменено, что намеренно не изменено, какие риски остались.
17. Работай инкрементально: после каждого завершенного логического блока, прошедшего локальные проверки, предлагай пользователю создать коммит.
18. Если объем изменений становится чрезмерным или включает несколько независимых блоков, обязательно предлагай разделить работу на 2+ отдельных коммита.
19. Не откладывай предложение коммита до конца большой задачи, если уже есть завершенный и проверенный инкремент.
20. Следуй `CommitPlan` из `docs/preproject/07-task-template.md` и фиксируй отклонения только по явному согласованию.
21. В commit message запрещены любые упоминания инструмента/ассистента (включая `Cursor`, `generated by Cursor`, `made with Cursor`, `co-authored-by Cursor`).
22. Commit message должен описывать только смысл изменения (Conventional Commit + why), без указания, каким инструментом выполнена работа.
